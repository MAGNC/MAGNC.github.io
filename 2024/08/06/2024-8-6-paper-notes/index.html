5G RAN

In 5G New Radio (NR) systems, scrambling is a crucial process applied to data and control information before modulation and transmission. Its primary purpose is to randomize the bit sequences, thereby enhancing signal integrity and security.

Purpose of Scrambling:

​ 1. Interference Mitigation: By randomizing bit patterns, scrambling minimizes the likelihood of repetitive patterns that can cause interference, thus improving overall network performance.

​ 2. Enhanced Security: Scrambling adds a layer of security by making it more challenging for unintended receivers to interpret the transmitted data without the correct descrambling sequence.

​ 3. Error Reduction: It helps in distributing errors more uniformly across the transmitted data, facilitating more effective error correction.

Scrambling Process in 5G NR:

The scrambling procedure involves combining the data sequence with a pseudorandom binary sequence (PRBS) using a bitwise modulo-2 addition (XOR operation). This PRBS is generated based on specific parameters, including the physical layer cell identity and the radio network temporary identifier (RNTI), ensuring that each user equipment (UE) has a unique scrambling sequence.

Detailed Steps:

​ 1. PRBS Generation: A PRBS is generated using a linear feedback shift register (LFSR) initialized with values derived from the cell identity and RNTI.

​ 2. Bitwise XOR Operation: The original data bits are combined with the PRBS using a bitwise XOR operation, resulting in a scrambled bit sequence.

​ 3. Transmission: The scrambled data is then modulated and transmitted over the air interface.

Descrambling at the Receiver:

Upon reception, the UE utilizes the same PRBS, generated using the known cell identity and RNTI, to perform a bitwise XOR operation with the received scrambled data, thereby retrieving the original bit sequence.

Head-Of-Line (HOL) problem

TCP’s HOL blocking problem: In TCP, if a packet is lost, all packets behind it must wait for the lost packet to be retransmitted and arrive in order, even if those packets have already been received correctly. This increases latency.

QUIC’s solution: QUIC handles multiple streams within a single connection independently. Each stream has its own flow control and delivery order, meaning a lost packet in one stream doesn’t block other streams, dramatically reducing latency.

Built-in Encryption (TLS Integration)

QUIC integrates the TLS 1.3 protocol natively, meaning all QUIC connections are encrypted by default. Unlike TCP, where encryption (TLS/SSL) is an optional layer added on top of the protocol, QUIC has encryption baked in from the start, ensuring confidentiality, integrity, and authentication in all communications.

This also reduces overhead and accelerates connection setup, as fewer round trips are required to establish both the connection and the encryption compared to TCP + TLS.

Reduced Latency: 0-RTT and 1-RTT Handshake

TCP handshake: In TCP, establishing a connection requires a three-way handshake, which typically involves at least one round-trip time (RTT) between client and server before any data can be sent. This adds latency, especially in high-latency networks.

QUIC handshake: QUIC can use a 1-RTT handshake or even a 0-RTT handshake under certain conditions. In the 0-RTT case, if a client has previously connected to a server, it can resume the connection and send data immediately without waiting for a full handshake, reducing latency drastically.

How 0-RTT works in QUIC

The basic idea behind 0-RTT connection resumption is that if the client and server had previously established a TLS connection between each other, they can use information cached from that session to establish a new one without having to negotiate the connection’s parameters from scratch.

0-RTT allows the client to start sending application data (e.g., HTTP requests) in the very first packet it sends to the server, without waiting for the entire handshake process to complete. Here’s how this works:

When a client has previously connected to a server and cached the server’s cryptographic parameters (e.g., the server’s certificate and encryption keys), it can use this information to resume the connection in subsequent interactions.

During the 0-RTT handshake, the client sends its initial data along with a cryptographic message (based on the cached information) that allows the server to verify the client and resume the session.

The server can immediately process this data and respond, allowing the client to avoid waiting for the handshake to complete.

Attack of the Clones

Data sent after the 0-RTT phase, meaning after the handshake has been completed, would still be safe though, as TLS 1.3 (and QUIC) will still perform the normal key exchange algorithm (which is forward secret) for data sent after the handshake completion.

However, application data sent during 0-RTT can be captured by an on-path attacker and then replayed multiple times to the same server. It seems that in many cases this is not a problem, as the attacker wouldn’t be able to decrypt the data, which is why 0-RTT connection. But in some cases, this can be dangerous.

How 1-RTT works in QUIC

The 1-RTT (One Round-Trip Time) handshake in QUIC is necessary to establish a secure connection when the client and server do not have prior session information or when 0-RTT is not feasible due to security concerns or other reasons. While 0-RTT allows the client to send data immediately, 1-RTT is needed for a full handshake to ensure full security, authentication, and encryption. Here’s a detailed explanation of why 1-RTT is used in QUIC:

1.Ensuring Security and Authentication:

1-RTT ensures that a proper handshake is completed between the client and server, establishing the following important aspects:

Encryption keys: The client and server negotiate fresh encryption keys during the handshake to secure the connection.

Authentication: The server (and optionally the client) is authenticated through cryptographic certificates, ensuring that both parties are who they claim to be.

Perfect Forward Secrecy (PFS): By generating new keys for each session, 1-RTT guarantees that past communications remain secure, even if long-term keys are compromised(泄露) in the future.

2.When 1-RTT is Needed：

1-RTT is used in several scenarios where 0-RTT cannot be applied:

No prior connection: If the client is connecting to the server for the first time, it has no prior session information (e.g., no cached session ticket or keys).

Expired session information: If a cached session ticket or other session information has expired or is no longer valid. The clinet must request a full 1-RTT handshake to renew the session.

Sensitive data transmission: Even if 0-RTT is available, certain sensitive data (e.g., login credentials, personal information) should only be sent after a 1-RTT handshake to ensure Perfect Forward Secrecy and strong encryption with fresh keys.

3.How 1-RTT Works in QUIC

1-RTT handshake is similar to handshake used in TLS1.3, but with optimizations specific to QUIC, here is the process:

ClientHello (CHLO):

The client sends an initial packet containing a ClientHello message, which includes its cryptographic parameters (e.g., supported encryption algorithms) and a key share for negotiating the encryption. In addition, if the client has no prior session information (e.g., no cached session ticket), this is a fresh connection, and no 0-RTT data is sent.

ServerHello (SHLO):

The server responds with a ServerHello message, agreeing on the encryption parameters and sending its own key share to complete the key exchange. The server also sends its certificate to authenticate itself.

Key Generation and Handshake Completion：

After receiving the server’s key share, both the client and server generate new session keys based on the exchanged key shares.

The handshake is considered complete when both the client and server agree on the encryption keys and the client has verified the server’s identity.———-> Complete!

4.Why 1-RTT is Still Faster than TCP

Although 1-RTT requires one round-trip to complete the handshake, it is still significantly faster than the traditional TCP + TLS handshake, there are reasons:

Combined Handshake: QUIC integrates both the transport-layer handshake (typically handled by TCP) and the cryptographic handshake (typically handled by TLS) into a single process, reducing the overall number of round-trips.

In TCP + TLS: A 3-way TCP handshake is followed by the TLS handshake, requiring multiple round-trips before data can be sent.(So it provides the possibility to implement Software defined network functions).

In QUIC 1-RTT: Only one round-trip is needed to establish both the transport connection and encryption, reducing the delay.

Fewer round-trips for data transmission: After 1-RTT completes, data can be transmitted immediately without any further delays, while in TCP, data transmission may still be delayed due to additional security or setup procedures.

5.Performance Balance Between 0-RTT and 1-RTT

0-RTT is optimized for speed but comes with some limitations, such as the risk of replay attacks and the lack of Perfect Forward Secrecy for the initial data sent.

So If the long-term keys (e.g., server’s private key) are compromised at a later point, past communications are still protected using PFS, as they were encrypted using session-specific keys that were never transmitted.

Zero Knowledge Proof

In a zero-knowledge proof, three main properties must be satisfied:

  1. Completeness: If the statement is true, an honest verifier will be convinced by an honest prover.
  2. Soundness: If the statement is false, no cheating prover can convince an honest verifier that it is true, except with some small probability.
  3. Zero-Knowledge: If the statement is true, the verifier learns nothing beyond the fact that the statement is true. Specifically, they gain no knowledge about the underlying secret or information.

Simplified Example (Metaphor: A Secret Door Puzzle)

Imagine a cave with two paths that split into two separate tunnels, meeting again at a door at the far end. The door can only be opened with a secret code. The goal is for Peggy (the prover) to convince Victor (the verifier) that she knows the code to open the door, without actually revealing the code itself.

  1. Peggy can enter the cave, take one of the two paths (either left or right), and reach the door.

  2. Victor stays outside the cave and doesn’t see which path Peggy took.

  3. Victor

    then asks Peggy to come out of a specific path (either left or right).

    • If Peggy knows the code, she can always go through the door and come out of the requested path.
    • If she doesn’t know the code, she can only come out of the correct path 50% of the time by guessing which path Victor will ask for.

  4. By repeating this process multiple times, Victor gains confidence that Peggy knows the secret code because she always exits from the correct path, proving she can open the door.

The key point is that Victor never learns what the actual code is but becomes convinced that Peggy knows it.

Detailed Explanation in Cryptographic Terms

Interactive ZKP:

Prover: The prover has a secret (let’s call it S) and wants to prove knowledge of S to the verifier without disclosing it.

Verifier: The verifier poses a series of challenges or questions to test the prover.

Responses: The prover responds to these challenges in such a way that the verifier becomes convinced that the prover knows S.

Non-Interactive ZKP:(Verification methods)

In some situations, interaction between prover and verifier is impractical. Non-interactive ZKPs exist, where the proof can be generated by the prover and then checked by the verifier without back-and-forth communication.

These types of ZKP systems often rely on public randomness or cryptographic hash functions.

ZKP Protocol Examples:

Fiat–Shamir Heuristic: A non-interactive ZKP method often used to convert interactive ZKP protocols into non-interactive ones. It replaces the verifier’s challenge with a deterministic value derived from hashing the prover’s message.

zk-SNARKs (Zero-Knowledge Succinct Non-Interactive Argument of Knowledge): zk-SNARKs enable the proof to be very short and quickly verifiable without interaction.

Applications

Blockchain and Cryptocurrencies: ZKP is crucial for privacy-focused cryptocurrencies like Zcash, which uses zk-SNARKs to allow transaction validation without revealing transaction details.

Authentication: ZKPs are used to create secure authentication systems. A user can prove they know a password without ever sending the password over the network, reducing the risk of interception.

Secure Voting: In a zero-knowledge-based voting system, a voter can prove they voted for a candidate without revealing their vote, maintaining privacy while ensuring the vote’s legitimacy.

Digital Identity: Zero-knowledge proofs can allow users to prove they meet certain conditions (such as age or citizenship) without revealing sensitive personal information. For example, you can prove you’re over 18 without sharing your birthdate.

Quantum Safe Direct Communication(QSDC)

Quantum Secure Direct Communication (QSDC) is a quantum communication protocol that enables two parties to exchange information securely and directly over a quantum channel, without the need for first establishing a shared secret key. Unlike Quantum Key Distribution (QKD), which is used to establish a secure key for later encryption of data, QSDC allows the actual message to be transmitted securely in real-time using quantum principles.

Key Features of QSDC

  1. Direct Secure Communication:

    • In QSDC, the message itself is encoded into quantum states and transmitted directly to the receiver, rather than just the key. This approach is particularly useful in scenarios requiring immediate secure data transmission.

  2. Security Based on Quantum Mechanics:

    • QSDC protocols rely on quantum mechanics to guarantee security, primarily through the no-cloning theorem and quantum measurement disturbance. Any eavesdropping on the quantum channel will introduce detectable errors in the transmission.

  3. Eavesdropping Detection:

    • Like QKD, QSDC can detect any interception attempts by an eavesdropper (often called Eve) because any measurement of the quantum states by Eve would disturb them, causing observable anomalies in the transmitted message.

  4. Message Encoding with Quantum States:

    • Information is often encoded in qubits, such as polarized photons, entangled states, or other quantum systems. The message is transmitted directly rather than being encrypted with a previously shared key.

Typical QSDC Protocols

QSDC protocols have been developed using different quantum principles and techniques. Commonly used QSDC protocols include:

  • Entangled QSDC Protocols: These rely on entanglement, where entangled photon pairs are shared between the sender and receiver. By measuring one of the entangled particles, the state of its partner is known, allowing for direct secure communication.
  • Single-Photon QSDC Protocols: Information is encoded into the polarization of single photons, and the protocol monitors the transmission for anomalies indicative of eavesdropping.

An example protocol is Ping-Pong QSDC:

  • The sender, Alice, prepares entangled pairs of photons and sends one photon from each pair to the receiver, Bob.
  • Bob either measures his photon or returns it to Alice, depending on the protocol phase.
  • After exchanges, Alice and Bob analyze their measurements to check for signs of eavesdropping and, if safe, use the results to convey information.

Advantages and Applications of QSDC

  • Real-Time Security: Since no separate key exchange step is necessary, QSDC allows secure communication in real-time, making it suitable for immediate high-security applications.
  • Quantum Network Communication: QSDC can be integrated into quantum networks, where direct secure communication is critical, such as military communications or secure financial transactions.
  • Enhanced Security Against Eavesdropping: QSDC provides a high level of security as any attempt to intercept the communication introduces errors detectable by the participants.

Challenges and Practical Considerations

  • Technology Requirements: QSDC requires precise quantum sources (like single-photon or entangled-photon sources), reliable quantum channels (such as fiber-optic links), and sensitive detectors to maintain high accuracy and low error rates.
  • Distance Limitations: As with other quantum communication methods, QSDC is affected by photon loss and noise over long distances, necessitating quantum repeaters for extended communication.

Conclusion

QSDC represents an advanced approach to secure communication by leveraging quantum principles to directly transmit messages. Its real-time security and robustness against eavesdropping make it a powerful tool in quantum communication, promising a new level of security for high-stakes, time-sensitive communication systems.

Compare QSDC and QKD

Quantum Secure Direct Communication (QSDC) and Quantum Key Distribution (QKD) are both quantum communication protocols that utilize quantum mechanics to achieve secure transmission. However, they differ fundamentally in their approach, purpose, and implementation. Here’s a comparison to highlight their key distinctions:

1. Purpose and Transmission Type

  • QSDC (Quantum Secure Direct Communication):

    • Directly transmits the message itself securely over a quantum channel.
    • Eliminates the need for a separate encryption step, as the message is embedded in the quantum states that are transmitted.
    • Best suited for applications where immediate secure message transfer is required, such as in high-security government or military communications.

  • QKD (Quantum Key Distribution):

    • Establishes a shared secret key between two parties, which is later used to encrypt and decrypt messages transmitted over a classical channel.
    • Ensures secure key exchange for subsequent encrypted communications, rather than transmitting the message directly over the quantum channel.
    • Ideal for applications needing long-term secure key establishment where the actual message transmission can occur over a classical network.

2. Protocol Structure and Process

  • QSDC:

    • Combines message encoding and secure transmission in a single quantum communication process.
    • Commonly uses entangled photons or single-photon sources to directly encode the message.
    • After confirming the quantum channel’s security, the sender and receiver exchange message-encoded qubits. If eavesdropping is detected, they abort or resend the message.

  • QKD:

    • Involves a two-step process:
      1. Quantum Key Exchange: Quantum states (typically single photons) are transmitted to establish a shared random key.
      2. Classical Communication: Afterward, the key is used to encrypt and decrypt messages over a classical channel.
    • Protocols like BB84 or E91 are standard for QKD, where the key is generated first, followed by message encryption with classical cryptographic methods.

3. Security Mechanism and Eavesdropping Detection

  • QSDC:

    • Directly Encodes Information: Since the message itself is encoded in the qubits, QSDC provides real-time detection of eavesdropping, as any interception will disturb the quantum states and introduce errors.
    • Continuous Security Checks: During the exchange, any interference or measurement by an eavesdropper (Eve) introduces detectable changes in the quantum states, ensuring that the presence of eavesdropping is known immediately.
    • No Key Storage Required: Security does not rely on post-key processing, reducing risks associated with key management.

  • QKD:

    • Indirect Eavesdropping Detection: QKD only detects eavesdropping during the key exchange phase. After the key is established, it’s used in classical encryption, so the actual message transmission is secure but outside quantum protections.
    • Key Management: The security of the message depends on secure key storage and management. If an attacker compromises the classical encryption key, the confidentiality of messages is at risk.
    • Privacy Amplification and Error Correction: In QKD, additional steps ensure that any potential partial information gained by an eavesdropper during the key exchange does not affect message confidentiality.

4. Efficiency and Practical Implementation

  • QSDC:

    • Complex Implementation: QSDC requires more complex setup and high fidelity in quantum state generation, detection, and measurement, as both message encoding and security checks are performed at the quantum level.
    • Lower Data Rates: Due to the sensitivity of direct quantum message encoding and transmission, QSDC protocols typically support lower data rates and are more challenging to implement over long distances.

  • QKD:

    • More Mature Technology: QKD is more established and has seen more real-world implementations due to the relative simplicity of exchanging only the key via quantum means.
    • Higher Data Rates: Since only the key is transmitted over the quantum channel and the actual message transmission occurs over classical networks, QKD can handle higher data rates and larger messages effectively.

5. Use Cases

  • QSDC:

    • Suited for immediate, real-time secure communication, such as government, military, or critical infrastructure applications.
    • Ideal in situations where both high-security requirements and real-time data exchange are necessary.

  • QKD:

    • Used widely for secure key exchange in encryption systems for industries such as banking, healthcare, and data centers.
    • Suitable for applications where secure key establishment is needed over long distances and where encrypted communication can occur later on a classical network.

Different Quantum Communication Technologies

No, QSDC (Quantum Secure Direct Communication) and QKD (Quantum Key Distribution) are not the only main quantum communication modes. While they are prominent for secure quantum communication, there are several other modes and protocols in quantum communication, each serving different purposes within a broader spectrum of quantum information transfer. Here are the main quantum communication modes in addition to QSDC and QKD:

1. Quantum Teleportation

  • Purpose: Quantum teleportation allows the transfer of quantum states from one location to another without physically moving the particle itself. Instead of transferring the particle, the information about the quantum state is transmitted.
  • Mechanism: Quantum teleportation requires two entangled particles shared between the sender (Alice) and the receiver (Bob). By performing a specific measurement on Alice’s particle and the particle whose state needs to be teleported, and then sending classical information about the measurement outcome to Bob, Bob can reconstruct the original quantum state.
  • Applications: Quantum teleportation is foundational for quantum networks and quantum computing infrastructure. It is particularly important for creating quantum internet connections and for transferring quantum states over long distances without direct transmission.

2. Quantum Dense Coding

  • Purpose: Quantum dense coding allows two classical bits of information to be transmitted by sending only one qubit, thus doubling the communication capacity compared to classical channels.
  • Mechanism: In dense coding, Alice and Bob share an entangled pair. Alice encodes her information by performing specific quantum operations (Pauli operations) on her qubit and then sends it to Bob. When Bob measures both qubits in his possession, he can infer the two classical bits of information Alice wanted to send.
  • Applications: Quantum dense coding is valuable for increasing data transfer efficiency in quantum communication networks, particularly in bandwidth-constrained environments.

3. Quantum Entanglement Distribution

  • Purpose: Entanglement distribution involves creating and distributing entangled states over long distances, which is a key resource for many quantum communication protocols.
  • Mechanism: Through methods like entanglement swapping or using quantum repeaters, entanglement can be shared over distances beyond what is possible with direct transmission.
  • Applications: This is essential for building large-scale quantum networks (like a quantum internet) and supports protocols like quantum teleportation, QKD, and distributed quantum computing.

4. Quantum Secret Sharing (QSS)

  • Purpose: Quantum secret sharing allows a secret message to be divided into multiple parts and shared among a group such that only a subset of recipients can reconstruct the original message when they collaborate.
  • Mechanism: QSS is based on entanglement and measurements across multiple participants, where quantum correlations enable the secret to be securely shared. In typical setups, each participant holds a part of an entangled state, and only specific measurements by multiple participants can reveal the message.
  • Applications: Quantum secret sharing is useful in secure multiparty communications and distributed access control, such as for managing classified information or in secure voting systems.

5. Quantum Digital Signatures (QDS)

  • Purpose: Quantum digital signatures authenticate messages in a way that prevents forgery and repudiation.
  • Mechanism: In QDS, the sender creates a signature based on quantum states and distributes parts of it to the recipients in a way that cannot be duplicated. Verification relies on quantum measurement outcomes that ensure message integrity and authenticity.
  • Applications: Quantum digital signatures are relevant for authenticating messages in secure communications, preventing tampering, and ensuring the validity of transmitted information in quantum networks.

The open issues in QKD network

Quantum Key Distribution (QKD) networks are advancing rapidly, but there are several open issues and challenges that need to be addressed to enable large-scale, reliable, and cost-effective deployment. Here’s a breakdown of the primary open issues in QKD networks:

1. Scalability and Network Integration

  • Issue: QKD networks face challenges in scaling to cover large geographic areas and integrating with existing classical networks.
  • Details: Traditional QKD is limited to point-to-point connections over a relatively short distance (usually up to 100–200 km in optical fiber), which limits its scalability in large networks. To overcome this, QKD networks require trusted nodes or quantum repeaters, which add complexity and potential vulnerabilities.
  • Potential Solutions: Research is focused on creating multi-hop QKD protocols and quantum repeaters that can extend the range without compromising security, as well as integrating QKD with Software-Defined Networks (SDNs) for flexible and scalable routing.

2. Distance Limitations and Quantum Repeaters

  • Issue: Photon loss over long distances limits QKD in optical fibers and free-space channels, necessitating intermediate nodes or repeaters.
  • Details: In fiber-based QKD, photon loss increases exponentially with distance due to absorption and scattering, making it difficult to maintain secure communication over distances beyond a few hundred kilometers. Current QKD networks rely on trusted relay nodes, which reintroduces security risks and reduces the overall quantum nature of the system.
  • Potential Solutions: Quantum repeaters are a potential solution but remain in the experimental stage. Repeaters that use entanglement swapping, quantum memory, and error correction could enable secure long-distance QKD but require significant technical advancements.

3. Standardization and Interoperability

  • Issue: QKD lacks standardized protocols and interfaces, which hinders interoperability between devices and network vendors.
  • Details: As QKD technology is developed by different vendors, devices may use incompatible protocols, key management schemes, and interfaces. This complicates network integration and makes it challenging to create unified QKD networks across multiple vendors and geographies.
  • Potential Solutions: Standardization bodies like the ITU-T, ETSI, and ISO are working on creating standards for QKD, including secure communication protocols, interfaces, and authentication mechanisms. However, reaching consensus and ensuring compatibility across all QKD implementations remains an open challenge.

4. Cost and Infrastructure Requirements

  • Issue: QKD systems are expensive and require specialized infrastructure, making large-scale deployment financially challenging.
  • Details: QKD requires expensive single-photon sources, detectors, and often dedicated fiber-optic infrastructure. In metropolitan QKD networks, dedicated fibers are often used, which can be cost-prohibitive for widespread implementation.
  • Potential Solutions: Developing cost-effective QKD hardware and exploring hybrid quantum-classical solutions where QKD runs alongside classical data on the same infrastructure may reduce costs. Ongoing research is also focused on integrating QKD with existing telecommunication networks to share infrastructure and reduce expenses.

5. Security and Trusted Nodes

  • Issue: Trusted relay nodes, required in current QKD networks to extend range, create potential points of vulnerability.
  • Details: Trusted nodes must decrypt and re-encrypt keys, making them vulnerable to physical or cyber attacks. This is problematic in a network setting, where multiple nodes might be needed to reach distant locations, potentially weakening end-to-end security.
  • Potential Solutions: Device-independent QKD and Measurement-Device-Independent (MDI) QKD are proposed to eliminate or reduce the dependency on trusted nodes, but they require advanced technology and are still in development.

6. Integration with Cryptographic Key Management

  • Issue: Integrating QKD with classical key management systems and cryptographic infrastructure is complex.
  • Details: Most existing key management infrastructures are built around public-key cryptography, and integrating QKD keys into these systems requires additional infrastructure and protocol changes.
  • Potential Solutions: Research into Quantum Key Management Systems (QKMS) is ongoing, aiming to develop systems that can effectively manage QKD keys and integrate them seamlessly into existing cryptographic protocols.

7. Performance in Real-World Conditions

  • Issue: QKD systems can suffer from decreased performance under real-world conditions such as weather variability, channel noise, and physical disturbances.
  • Details: Environmental factors like weather affect free-space QKD, while fiber networks can experience noise from neighboring classical channels. Temperature changes and physical disruptions can also impact the stability of photon sources and detectors.
  • Potential Solutions: Developing more robust QKD hardware, error correction protocols, and adaptive modulation techniques can help QKD systems perform reliably in diverse and challenging environments.

8. Quantum Network Security Beyond Eavesdropping

  • Issue: While QKD addresses eavesdropping, other security threats like denial-of-service (DoS) attacks and node tampering are still concerns.
  • Details: QKD protocols are designed to detect and prevent eavesdropping, but they do not inherently prevent attacks on network infrastructure, such as DoS attacks on QKD channels or tampering with nodes.
  • Potential Solutions: Integrating classical cybersecurity measures with QKD networks, such as intrusion detection systems (IDS) and physical security protocols, can mitigate non-eavesdropping-related threats.

9. Practical and Theoretical Limitations of Quantum Protocols

  • Issue: QKD protocols are limited by theoretical bounds, such as error thresholds, and practical limitations in real-world deployments.
  • Details: Quantum error correction and security thresholds are constrained by factors like photon loss and detector efficiency, limiting the achievable security rates and distances. For example, finite-key analysis shows that a high-security QKD link needs a large number of exchanged qubits, which is challenging to maintain over long distances.
  • Potential Solutions: Developing error-tolerant QKD protocols and efficient finite-key analysis methods can improve the robustness and reliability of QKD, even under realistic constraints.

10. Development of Quantum Repeaters and Quantum Internet

  • Issue: True long-distance QKD and a quantum internet require functional quantum repeaters, which are still under development.
  • Details: Quantum repeaters, essential for extending QKD over long distances, rely on advanced technologies like quantum memory and entanglement swapping, which are not yet fully operational at scale. Building a quantum internet where QKD networks interconnect globally also requires solving these technological challenges.
  • Potential Solutions: Ongoing research in quantum repeater technology, quantum entanglement distribution, and quantum internet architectures is expected to gradually address this challenge, paving the way for scalable, interconnected QKD networks.

Conclusion

These open issues in QKD networks underscore the complexity and challenges in deploying quantum-secure communication at a large scale. While QKD has demonstrated significant potential for secure communication, addressing these issues will be essential for enabling practical, efficient, and widespread use of QKD networks. Advances in quantum repeaters, standardization, cost reduction, and robust hardware will play critical roles in the next stages of QKD’s development toward a truly scalable and integrated quantum network.

Existing projects about QKD

The Quantum Space Satellite, aka Quantum Experiments at Space Scale (QUESS).

QUESS is a proof-of-concept mission designed to facilitate quantum optics experiments over long distances to allow the development of quantum encryption and quantum teleportation technology. Quantum encryption uses the principle of entanglement to facilitate communication that can absolutely detect whether a third party has intercepted a message in transit thus denying undetected decryption.

It will be launched in July 2016. Chief scientist Pan Jianwei remarks that QUESS will complete China’s growing quantum communications network, which includes a 2,000-kilometer-long network between Beijing and Shanghai. QUESS’s function is to test the phenomena of quantum entanglement. Operated by the China Academy of Sciences, this 500kg satellite contains a quantum key communicator, quantum entanglement emitter, entanglement source, processing unit, and a laser communicator.

In addition to its own efforts, Chinese scientists are teaming up with their European counterparts on other quantum technologies such as photon teleportation, transmission error reduction and random number generators. If QUESS is successful, China will build an Asian-European quantum key distribution network by 2020, to be followed by a global quantum communications network in 2030.

QKD has been attempted on Earth, both with direct line-of-sight between two observatories, and using fibre optic cables to transmit the photons. However, fiber optics and the atmosphere both cause scattering, which destroys the entangled state, and this limits the distance over which QKD can be carried out. Sending the keys from an orbiting satellite results in less scattering, which allows QKD to be performed over much greater distances.

In addition, QUESS could test some of the basic foundations of quantum mechanics. Bell’s theorem says that no local hidden-variable theory can ever reproduce the predictions of quantum physics, and QUESS was able to test the principle of locality over 1,200 km (750 mi)

Eagle-1 quantum key distribution satellite

In 2024, ESA intends to launch the Eagle-1 quantum key distribution satellite, with a goal similar to that of the Chinese QUESS. It will be part of the development and deployment of the European Quantum Communication Infrastructure (EuroQCI).
